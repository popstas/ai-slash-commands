# Build the `do` skill (TODO.md automation orchestrator)

## Context
`docs/TODO.md` asks for a new Claude Code skill, `do`, that turns a project's `docs/TODO.md`
into an autonomous "when there's enough work queued, nudge me to plan & code it" loop:

- A cron-driven script decides (daily) whether `docs/TODO.md` is "ready" for `ralphex-adopt`,
  and if so sends a Telegram message with a link/command that opens Claude in the project and
  runs `/ralphex:ralphex-adopt docs/TODO.md`.
- Run manually, `/do` evaluates the task list, can run ralphex-adopt → ralphex, and edits the
  task list (`do add`, `do remove`). Commit prefix for task-list changes: `todo:`.
- Telegram delivery adapts: inside a Telegram-capable agent (claudeclaw/openclaw/hermes) it uses
  the agent's reply tool; standalone/cron it shells out to a bundled `telegram-send` CLI.

This repo (`ai-slash-commands`) is a Node.js generator: `prompts/*.md` → `dist/**` → installed to
six editor homes. Skills live separately in `skills/<name>/SKILL.md` (only `commit` exists today and
is **not yet wired into the build**). This task also wires `skills/` into gen + install.

### Decisions locked with user
- **Readiness flow**: the Python script decides fully via heuristics and sends Telegram directly
  (runs ~daily). Launching an LLM agent (`codex` by default) is an opt-in mode (`DO_LAUNCH_AGENT=1`).
- **telegram-send**: bundle a self-contained POSIX wrapper (`curl` + Bot API), auto-installed to PATH.
- **Adopt link**: include **both** a copy-paste shell command and a best-effort custom-scheme URL.
- **Scope**: also wire `skills/` into `gen.mjs` / `install-configs.mjs` (covers `do` and `commit`).

### Key files
- `scripts/gen.mjs` — build pipeline (`prompts/*.md` → `dist/<target>/...`). `TARGETS` map, `generate()`.
- `scripts/install-configs.mjs` — `dist/**` → home dirs. `DEST`/`SRC` maps, `copyAll()`, `install()`.
- `scripts/uninstall.mjs`, `scripts/cli.mjs` — keep consistent with new skill outputs.
- `skills/commit/SKILL.md` — existing skill format reference (`--- name / description ---` frontmatter).
- `prompts/*.md` — command format reference (first line `# name - description`).
- No test runner exists yet (`package.json` has no `test` script).

## Development Approach
- **Testing approach**: Regular (code first, then tests) — confirmed.
- Stdlib-only Python (no venv/deps) and POSIX `sh` so cron has zero install friction.
- Every task ends by writing/updating tests and running them green before the next task.
- Add a `test` script: `node --test` for JS, `python3 -m unittest` for the Python helper, plus a
  shell test for `telegram-send` using a `DRY_RUN` mode + fake `curl` on PATH.
- Keep backward compatibility: prompts pipeline behavior unchanged; skills are additive.

## Testing Strategy
- **Unit (JS)**: `node --test` for new gen/install skill helpers (SKILL.md→shim conversion, dir copy).
- **Unit (Python)**: `python3 -m unittest` for readiness heuristics, debounce/state, message build.
- **Shell**: `telegram-send` `DRY_RUN=1` prints the request it would send; assert URL/payload.
- No e2e/UI in this project.

## Technical Details
- **Skill → command shim**: `skillToCommand(name, body)` = strip leading `---...---` frontmatter,
  prepend `# ${name} - ${description}\n\n`, reuse existing per-target write logic (incl. antigravity
  frontmatter form).
- **State file**: `DO_STATE_DIR/<slug(project_dir)>.json` = `{ "hash": "...", "notified_at": "ISO" }`.
- **Task-unit count**: lines matching `^\s*[-*] (\[[ xX]\] )?\S` plus `^#{1,3} \S` non-title headings.
- **telegram-send contract**: `telegram-send "msg"` (or stdin); env `TELEGRAM_TOKEN`,`TELEGRAM_CHAT_ID`;
  `DRY_RUN=1` prints request; exit 2 on missing creds, 1 on curl failure.
- **Environment detection (in SKILL.md, for LLM execution)**: telegram MCP `reply` tool / inbound
  `<channel source="telegram">` ⇒ telegram-agent (reply via tool); **hermes** ⇒ offer to launch a
  terminal (or follow project `CLAUDE.md`/`AGENTS.md` handoff instructions if present); else (cron/
  standalone) ⇒ CLI `telegram-send`.
